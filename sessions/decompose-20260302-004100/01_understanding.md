## Goal

Evolve slipstream from a pure in-memory text editor into a **file session coordinator** that adds a format registry, format-aware open guidance, a global session tracker, and coordination commands — without changing any existing text editing behavior.

---

## Components

### Component 1: Format Registry (`slipstream-daemon`)

**What it is:** A TOML configuration file (`~/.config/slipstream/formats.toml`) with a compiled-in default set. Maps file extensions to handler descriptors. Two handler types: full (session_open/session_save/help_tool) and advisory-only (guidance text, no session lifecycle).

**New files to create:**
- `crates/slipstream-daemon/src/registry.rs` — `FormatRegistry`, `HandlerConfig`, `AdvisoryConfig`, TOML parsing, built-in defaults
- `~/.config/slipstream/formats.toml` — user-configurable file (created at first run or by documentation, not by code)

**Files to modify:**
- `crates/slipstream-daemon/Cargo.toml` — add `toml = "0.8"` dependency (TOML deserialization)
- `crates/slipstream-daemon/src/main.rs` — load registry at daemon startup, pass to serve/handler
- `crates/slipstream-daemon/src/lib.rs` — expose registry module, thread `Arc<FormatRegistry>` through `serve()`
- `crates/slipstream-daemon/src/handler.rs` — pass registry to `dispatch()`, use in `handle_session_open()`

**Key design details:**
- `{path}` template variable in `session_open` string
- `advisory = true` for Makefile/Dockerfile: loaded as text but with guidance appended
- Extensions are matched by the file extension of the path being opened (lowercased)
- Built-in defaults compiled in via `include_str!` or literal Rust struct initialization (no file needed at startup)

### Component 2: Format-Aware Open

**What it is:** Modification to the `handle_session_open` handler to consult the registry before loading files as text buffers.

**Behavior:**
- Extension not in registry (or `native`) → existing path (load text buffer, return `SessionOpenResult`)
- Extension matches full handler → return `ExternalHandlerResult` JSON, register in session tracker, do NOT open a text buffer
- Extension matches advisory handler → load as text buffer AND include guidance in response
- Unknown extension → fall through to existing text path

**Response shapes (new):**
```json
// External handler response
{
  "status": "external_handler",
  "path": "...",
  "handler": "sheets",
  "description": "...",
  "instructions": { "open": "...", "save": "...", "help": "...", "examples": [...] },
  "tracking_id": "ext-NNN"
}

// Advisory response (augments existing SessionOpenResult)
{
  "status": "advisory",
  "path": "...",
  "handler": "make",
  "guidance": "...",
  "loaded_as_text": true
}
```

**Files to modify:**
- `crates/slipstream-daemon/src/handler.rs` — `handle_session_open()` checks registry per file path
- `crates/slipstream-daemon/src/types.rs` — add `ExternalHandlerResult`, `AdvisoryInfo`, update `SessionOpenResult` to carry optional advisory fields

**Note:** `session.open` currently accepts a list of files. When the list is mixed (some text, some external), the design doc implies guidance is returned per-file. This needs a decision (see Ambiguities section).

### Component 3: Session Tracker (`slipstream-daemon`)

**What it is:** A new `Coordinator` struct (or `SessionTracker`) holding a `DashMap<PathBuf, TrackedFile>`. Distinct from `SessionManager` — it is a coordinator-level structure above the text-buffer layer.

**Data model (from doc):**
```rust
pub struct TrackedFile {
    pub path: PathBuf,
    pub canonical_path: PathBuf,
    pub handler: HandlerType,
    pub state: FileState,
    pub tracking_id: String,
    pub registered_at: Instant,
    pub last_activity: Instant,
}

pub enum HandlerType {
    Native { session_id: SessionId },
    External { handler_name: String },
    Advisory { handler_name: String, session_id: SessionId },
}

pub enum FileState {
    Clean,
    Dirty { edit_count: usize },
    Flushed,
    ExternallyManaged,
    Closed,
}
```

**New files to create:**
- `crates/slipstream-daemon/src/coordinator.rs` — `Coordinator` struct, `TrackedFile`, `HandlerType`, `FileState`, `SessionDigest`, `DigestEntry`, all tracker operations

**Files to modify:**
- `crates/slipstream-daemon/src/lib.rs` — expose `coordinator` module
- `crates/slipstream-daemon/src/main.rs` — instantiate `Arc<Coordinator>`, thread through `serve()`
- `crates/slipstream-daemon/src/handler.rs` — register native files on `session.open`, update state on write/str_replace/flush/close

**Key detail:** The tracker must be kept in sync with native sessions. When `session.flush` completes, tracked files move from `Dirty` to `Flushed`. When `session.close` completes, tracked files move to `Closed`. This requires the handler to update the tracker after every mutation.

**Tracking ID generation:** The doc shows `"ext-001"` style IDs. Need a counter (e.g., `AtomicU64`) or UUID. The doc shows sequential numeric suffixes which suggests a counter.

### Component 4: Global Session Digest

**What it is:** A formatted summary of all tracked files, appended to mutation responses.

**Data model:**
```rust
pub struct SessionDigest {
    pub total_tracked: usize,
    pub native_count: usize,
    pub native_dirty: usize,
    pub external_count: usize,
    pub files: Vec<DigestEntry>,
}

pub struct DigestEntry {
    pub path: String,      // relative to CWD if possible
    pub handler: String,
    pub state: String,     // human-readable: "3 edits pending", "clean", etc.
}
```

**When included:**
- After every mutation response: `session.open`, `file.write`, `file.str_replace`, `session.flush`, `session.close`, `coordinator.register`, `coordinator.unregister`
- On `coordinator.status` explicitly
- NOT on `file.read` (read-only, keep lean)
- NOT on `cursor.move`

**Files to modify:**
- `crates/slipstream-daemon/src/coordinator.rs` — `build_digest()` method
- `crates/slipstream-daemon/src/handler.rs` — all mutation handlers append digest to their response

**Implementation note:** The existing `Response::ok(id, result)` takes a `Serialize` value. Appending a digest field means either (a) wrapping the result type in a new struct that includes `session_digest: Option<SessionDigest>`, or (b) serializing to `serde_json::Value` and merging. Option (a) is cleaner but requires touching all result types in `types.rs`. Option (b) is more pragmatic. This is an implementation decision to flag.

### Component 5: Coordination Commands

**What it is:** Four new JSON-RPC methods added to `dispatch()` in `handler.rs`.

**New methods:**
- `coordinator.status` — full digest + native sessions + external registrations + warnings
- `coordinator.register` — add an external file to the tracker (LLM calls after opening in FCP)
- `coordinator.unregister` — remove a file from the tracker (by tracking_id)
- `coordinator.check` — pre-flight check for a named action (currently only "build"), returns warnings + suggestion

**Request/response shapes (from doc):**
```json
// coordinator.register
params: { "path": "...", "handler": "sheets" }
result: { "tracking_id": "ext-001", "status": "registered" }

// coordinator.unregister
params: { "tracking_id": "ext-001" }
result: {}

// coordinator.check
params: { "action": "build" }
result: { "warnings": [...], "suggestion": "..." }

// coordinator.status
result: {
  "tracked_files": [...],
  "native_sessions": [...],
  "external_registrations": [...],
  "warnings": [...]
}
```

**Files to modify:**
- `crates/slipstream-daemon/src/handler.rs` — add four new `match` arms in `dispatch()`, implement handlers
- `crates/slipstream-daemon/src/types.rs` — add `CoordinatorRegisterParams`, `CoordinatorUnregisterParams`, `CoordinatorCheckParams`, corresponding result types
- `crates/slipstream-daemon/src/coordinator.rs` — implement `check_action()` logic

## Ambiguities and Flags

### Flag 1: Mixed file list in `session.open` (UNCLEAR)

`session.open` takes a list of files. If the list contains a mix of text files and `.xlsx` files, what happens?

**Option A:** Error — session.open is text-only; external files must be handled separately.
**Option B:** Partial response — text files get a session_id, external files get guidance responses, both in the same response.
**Option C:** The call is split — text files open normally, external files get guidance, response is a map from path to result.

The doc's behavior matrix discusses individual files but doesn't address the mixed list case explicitly. The current `SessionOpenResult` has a `files: HashMap<PathBuf, FileInfo>` for all opened files with a single `session_id`. Option A is simplest and avoids redesigning the response shape. Recommend: **reject external files in session.open with a clear error pointing to guidance, rather than silently mixing response types.** Needs design decision.

### Flag 2: Digest on mutation responses — response shape change

Current mutation responses are strongly typed structs (`FileWriteResult`, `SessionFlushResult`, etc.). Adding a `session_digest` field requires either:
- (A) Wrapping every result type in a generic envelope: `struct Response<T> { result: T, session_digest: Option<SessionDigest> }` — clean but touches all types
- (B) Serializing to `serde_json::Value` and inserting the field — pragmatic but loses type safety

The design doc doesn't specify the shape change mechanism. This decision affects whether the CLI and MCP clients need updating to parse the new field. Both clients currently only care about specific result fields, so Option B is viable. Recommend flagging for decision.

### Flag 3: `coordinator.check` action vocabulary

The doc only shows `"build"` as a supported action. Is this the only supported action? Are unsupported action names errors or no-ops? Should the implementation be open-ended (any unknown action triggers generic "flush all dirty files" advice) or strict (only "build" is valid)?

### Flag 4: Tracking ID generation strategy

The doc shows `"ext-001"` style IDs. A `DashMap`-based tracker with concurrent registration needs a thread-safe counter (`AtomicU64`) or UUID. The sequential style implies a counter, but a UUID would be simpler. This is a minor implementation choice, but should be consistent with the tracking_id shown in `coordinator.unregister` params.

### Flag 5: Session Tracker and SessionManager synchronization

The `SessionManager` in `slipstream-core` is unaware of the tracker. The tracker lives in `slipstream-daemon`. The handler must manually update tracker state after every `SessionManager` operation. This creates a two-phase update pattern that could drift (e.g., if a session is swept by the inactivity sweeper, the tracker won't know).

The sweeper in `main.rs` calls `sweep_expired()` independently. The tracker needs to be updated when sessions expire. The current sweeper loop in `main.rs` ignores sweep results beyond logging. This will need to also remove swept sessions from the tracker.

### Flag 6: `file.write` on an externally-tracked path — where does the warning appear?

The doc says "warn that the file is managed by another tool". But `file.write` currently returns `FileWriteResult { edits_pending }`. The warning would need to be either:
- Embedded in the `session_digest` field (the digest shows conflicting state)
- A separate `warnings` field in the response
- A separate `conflict_advisory` field

The doc doesn't specify the exact response field. Since the session digest is already planned to be appended to mutations, the warning could be part of the digest. But it's advisory (non-blocking), so the write still succeeds and returns normally. Recommend: put the advisory in the session_digest, not as an error.

### Flag 7: `slipstream-daemon` does not currently depend on `dashmap` directly

The `DashMap` used in `SessionManager` is in `slipstream-core`. The new `Coordinator` struct in `slipstream-daemon` will also need `DashMap`, which requires adding it to `slipstream-daemon/Cargo.toml`.
